# System Architecture

## High-level Flow

`User -> Desktop App -> Agent Core -> Tools Layer -> Google Workspace APIs`

Mọi action cần phê duyệt sẽ đi qua nhánh HITL:

`Agent Core -> HITL Queue -> Approval UI (Desktop) -> Execute/Cancel`

## Thành phần chính

### 1) Desktop App (Win/Mac/Linux)

Vai trò:

- Chat interface để user giao tiếp với agent.
- Approval center để duyệt các action pending.
- Notification system cho yêu cầu phê duyệt theo thời gian thực.

Tech gợi ý:
Frontend: Tauri v2 + React/Next.js + Tailwind CSS + shadcn/ui 
- **Shell**: Tauri.
- **UI**: React.

### 2) Agent Core

Vai trò:

- Nhận intent từ user input.
- Lập kế hoạch các bước thực thi.
- Chọn tool và điều phối execution.
- Gán mức rủi ro cho từng action để quyết định có cần HITL hay không.

Module đề xuất:

- **Planner (LLM)**
- **Executor**
- **Policy & Risk Evaluator**
- **Memory**

### 3) Tools Layer (quan trọng)

Hệ thống ưu tiên sử dụng **LangChain Google Workspace Tools** (từ các package chính thức) để tận dụng sự ổn định và cộng đồng. Chỉ xây dựng custom adapter khi LangChain chưa hỗ trợ tính năng đó (ví dụ: các thao tác phức tạp trên Google Docs).

Chuẩn hóa thao tác Workspace thành các hàm tool nhất quán, ví dụ:

- `gmail.read_email()`
- `gmail.create_draft()`
- `gmail.send_email()`
- `sheets.read_range()`
- `sheets.write_range()`
- `drive.search_file()`
- `drive.upload_file()`
- `drive.share_file()`

Tools Layer chịu trách nhiệm:

- Mapping từ action schema sang Google API call cụ thể.
- Validate input trước khi gọi API.
- Ghi log request/response cho audit.

### 4) HITL Engine

Flow chuẩn:

1. Agent tạo `ActionProposal`.
2. Risk evaluator xác định mức Low/Medium/High.
3. Nếu cần duyệt: đẩy proposal vào `Approval Queue`.
4. Desktop UI hiển thị thông tin + diff tác động.
5. User chọn Approve / Edit / Reject.
6. Engine phát lệnh execute hoặc cancel, đồng bộ trạng thái.

## Luồng dữ liệu logic

1. User nhập yêu cầu: "Gửi báo cáo tuần cho team từ sheet X".
2. Agent phân rã task: đọc sheet -> tạo nội dung email -> gửi email.
3. Action `send_email` được đánh dấu High risk.
4. UI hiển thị nội dung email cho user duyệt ngay trên app desktop.
5. Sau Approve, system mới gọi Gmail API send.

## Non-functional requirements

- **Cross-platform parity**: hành vi nhất quán trên Windows/macOS/Linux.
- **Observability**: trace theo `task_id`, `action_id`, `approval_id`.
- **Reliability**: retry policy cho lỗi network/API quota.
- **Security-first**: nguyên tắc least privilege theo OAuth scope.
