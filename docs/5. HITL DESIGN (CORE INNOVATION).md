# HITL Design (Core Innovation)

## 1) HITL Approval Gate là trung tâm của sản phẩm

Thay vì để agent tự động thực thi toàn bộ, hệ thống bắt buộc đi qua "cổng phê duyệt" cho các action nhạy cảm. Cổng này chạy trực tiếp trong desktop app, không phụ thuộc web dashboard.

## 2) Risk Levels

| Level | Ví dụ | HITL |
|---|---|---|
| Low | `read_email`, `list_drive_files` | Khong bat buoc |
| Medium | `create_draft`, `update_sheet_range` | Optional theo policy |
| High | `send_email`, `share_drive_external`, `delete_file` | Bat buoc |

## 3) Approval Payload

Mỗi đề xuất gửi tới user cần đủ ngữ cảnh:

```json
{
  "approval_id": "apr_001",
  "action_id": "act_001",
  "tool": "gmail.send_email",
  "summary": "Gui bao cao tuan cho team",
  "preview": {
    "to": ["abc@gmail.com"],
    "subject": "Weekly Report",
    "body_excerpt": "Hello team, ..."
  },
  "risk_level": "high",
  "reason": "External send",
  "expires_at": "2026-04-01T10:30:00Z"
}
```

## 4) Triển khai kỹ thuật (Technical Implementation)

Hệ thống sử dụng **Cách B: HumanInTheLoopMiddleware** từ LangChain/LangGraph để thiết lập các chính sách bảo mật (Security Policies) mà không cần can thiệp sâu vào code của từng công cụ (Tool).

### Cấu hình Middleware
Middleware sẽ tự động chặn các hành động (intercept) dựa trên danh sách các công cụ nhạy cảm được cấu hình:

```python
from langchain.agents.middleware import HumanInTheLoopMiddleware
from langgraph.checkpoint.memory import InMemorySaver

# Khởi tạo middleware điều phối phê duyệt
hitl_middleware = HumanInTheLoopMiddleware(
    interrupt_on={
        "gmail.send_email": {
            "allowed_decisions": ["approve", "edit", "reject"], # Cho phép sửa nội dung
        },
        "drive.delete_file": True, # Bắt buộc phê duyệt, không cho sửa
        "sheets.update_range": {
            "allowed_decisions": ["approve", "reject"],
        },
        "gmail.read_email": False, # Cho phép tự động thực thi (Low Risk)
    },
    description_prefix="Hành động đang chờ bạn xác nhận",
)

# Tích hợp vào Agent
agent = create_agent(
    model=llm,
    tools=gworkspace_tools,
    middleware=[hitl_middleware],
    checkpointer=InMemorySaver(), # Bắt buộc để lưu trạng thái khi tạm dừng
)
```

### Quyền hạn của người dùng (Decision Types)

User có 3 lựa chọn chính trên giao diện Desktop App:

1.  **Duyệt (Approve):** Cho phép Agent thực thi công cụ với các tham số ban đầu (ví dụ: gửi đúng nội dung email đã soạn).
2.  **Sửa & Duyệt (Edit & Approve):** Chỉnh sửa các tham số đầu vào của công cụ (ví dụ: sửa lại tiêu đề hoặc người nhận email) trước khi cho phép thực thi.
3.  **Từ chối (Reject):** Ngăn chặn hành động. Agent sẽ nhận được thông báo "Hành động bị từ chối bởi người dùng" và sẽ điều chỉnh kế hoạch tiếp theo (ví dụ: đề xuất một phương án khác).

## 5) UI Flow trên Desktop

1. Agent tạo action proposal.
2. HITL engine đưa vào queue.
3. Desktop app hiện notification native + badge pending.
4. User mở panel duyệt, xem preview tác động.
5. User quyết định Approve/Edit/Reject.
6. System cập nhật trạng thái và phản hồi lại chat timeline.

## 6) Advanced HITL Capabilities (phase sau)

- **Batch approval** cho nhiều action tương tự.
- **Auto-approve rules** (vd: chỉ gửi email nội bộ domain công ty).
- **Confidence threshold** để điều chỉnh khi nào cần duyệt.
- **Delegated approval** cho manager/team lead trong môi trường doanh nghiệp.

## 7) Audit & Compliance

Mọi phê duyệt phải truy vết được:

- Ai duyệt, duyệt khi nào, duyệt cái gì.
- Nội dung trước/sau khi chỉnh sửa.
- Kết quả thực thi sau phê duyệt.

Điều này tạo khác biệt lớn giữa "assistant trả lời" và "agent được phép hành động".
